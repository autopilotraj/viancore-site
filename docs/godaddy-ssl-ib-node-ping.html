IB Node Ping Failure After GoDaddy SSL Certificate Renewal

A GoDaddy certificate chain change — from the legacy G2 hierarchy to the new R1 hierarchy — caused Integration Broker node ping to fail silently after a routine SSL renewal. Here is what happened and how it was resolved.

What Happened

After renewing an SSL certificate on WebLogic, PeopleSoft Integration Broker node ping began failing with error 158,10836 — General Connection Failed. Network connectivity was confirmed working. The real problem was that the renewed GoDaddy certificate was issued under a new hierarchy (R1) not yet trusted in PeopleSoft Digital Certificates. Importing the new root and intermediate chain resolved the issue immediately.

01 · Error Observed

What PeopleSoft Reported

PeopleSoft displayed the following error when attempting a node ping through the Integration Broker Gateway:

Integration Gateway: General Connection Failed (158,10836)

PeopleSoft's own guidance for this error lists possible causes including a bad Gateway URL, a Sync Service timeout, or a Java exception on the application server. In this case, none of those were the actual cause.

02 · Initial Validation

What curl Told Us — And What It Didn't

The application server was tested using curl against the load balancer Gateway URL. The test succeeded, confirming DNS, routing, and firewall paths were all working.

Important distinction curl success only proves network reachability. It does not prove that PeopleSoft's Java runtime and Integration Broker SSL validation stack trust the certificate chain. These are different trust stores operating at different layers.

PeopleSoft node ping runs through the application server, the Java runtime, Integration Broker, and the PeopleSoft Digital Certificates trust configuration. All four layers must succeed for a node ping to complete.

03 · Root Cause

The Certificate Chain Changed

Comparing the old working certificate against the newly renewed certificate revealed a different GoDaddy hierarchy:

Old Chain — Previously Working
Go Daddy Root CA — G2
↓
Go Daddy Secure CA — G2
↓
Server / Leaf Certificate
New Chain — After Renewal
Go Daddy Root CA — G2
↓
GoDaddy TLS Root CA — R1 New
↓
GoDaddy TLS Intermediate CA DV — R1v1 New
↓
Server / Leaf Certificate

The renewed certificate introduced two additional components — the R1 root and the R1v1 intermediate — that were not present in PeopleSoft Digital Certificates. Because PeopleSoft could not build a complete trust path to the leaf certificate, the SSL handshake failed.

Why this surprised the team The assumption was that renewing from the same vendor would reuse the same root and intermediate hierarchy. That assumption is not safe. GoDaddy officially transitioned DV certificate issuance to the R1 hierarchy — any renewal after this transition produces a certificate with a different chain.
GoDaddy — Official Vendor Context

GoDaddy has formally migrated public DV certificate issuance to the GoDaddy TLS Root CA – R1 hierarchy using the GoDaddy TLS Intermediate CA DV – R1v1 issuing CA. This is a deliberate architectural change, not a temporary variation.

During the transition, GoDaddy maintains a cross-signed trust path from R1 back to G2, which allows browsers to validate the chain automatically. However, systems like PeopleSoft that maintain their own internal trust store must be updated manually — they do not benefit from the cross-signed path.

Certificate bundle files (including the R1 root and R1v1 intermediate) are available at certs.godaddy.com.

04 · Observation

Key Size Change — 2048 to 4096 Bit

The old certificate used a 2048-bit RSA key. The renewed certificate used a 4096-bit RSA key, reflecting stronger current security standards.

Common misdiagnosis risk The key size change is real but was not the cause of the failure. The failure was entirely due to the missing certificate chain in PeopleSoft Digital Certificates. Do not conflate RSA key size with certificate chain trust — they are separate concerns.
05 · Resolution

What Fixed It

The new GoDaddy certificate bundle was downloaded and compared against the certificates present in PeopleSoft Digital Certificates. The following were imported:

Certificates Imported GoDaddy TLS Root CA – R1 (new root) · GoDaddy TLS Intermediate CA DV – R1v1 (intermediate) · any additional intermediates present in the bundle

After importing the complete chain, PeopleSoft validated the WebLogic SSL certificate successfully. Integration Broker node ping completed. No Gateway URL changes and no load balancer routing changes were required.

06 · Future Reference

SSL Renewal Checklist for PeopleSoft / WebLogic

Use this checklist for any future WebLogic or PeopleSoft SSL renewal to prevent the same issue from recurring.

  • 1Capture the old certificate hierarchy before renewal begins.
  • 2Capture the new certificate hierarchy immediately after renewal.
  • 3Compare old and new root and intermediate certificate names explicitly — do not assume they match.
  • 4Confirm whether the chain changed hierarchy (e.g., G2 to R1) or added new intermediates.
  • 5Download the official certificate bundle from the CA (e.g., certs.godaddy.com).
  • 6Import any new required root and intermediate certificates into PeopleSoft Digital Certificates.
  • 7Confirm WebLogic or the load balancer is presenting the full certificate chain.
  • 8Restart required PeopleSoft / WebLogic services if applicable.
  • 9Test Gateway Ping from the Integration Broker Gateway configuration page.
  • 10Load Gateway Connectors and confirm they respond correctly.
  • 11Test Node Ping from the Integration Broker Network configuration page.
  • 12Save screenshots and logs as change evidence before closing the record.

08 · Lessons Learned

Key Takeaways

Certificate renewal is not just a file swap

Even when using the same certificate authority, the renewed certificate may be issued under a different root or intermediate hierarchy. Compare chains explicitly every time.

curl success does not mean PeopleSoft SSL success

OS-level network reachability and PeopleSoft's Java-based SSL trust validation are independent. Always complete a full node ping test as part of SSL renewal validation.

GoDaddy's R1 migration is permanent and ongoing

Any environment that has not yet renewed its certificate under the R1 chain will encounter this issue on next renewal. Proactive preparation eliminates unplanned downtime.

PeopleSoft Digital Certificates requires manual trust chain updates

Unlike browsers that benefit from GoDaddy's cross-signed trust path automatically, PeopleSoft's internal trust store must be managed explicitly on every renewal.
